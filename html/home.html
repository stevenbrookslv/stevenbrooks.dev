    clChat

    Developed using: Python

    clChat is a small program that allows for instant messaging via a command line on Linux or Mac computers, with no need for a central server or relay point. It allows for communication on local area networks OR across the internet as a whole. The local area network communication is fairly simple in that the only requirement is that users be on the same net. For communication across the internet however, a router that is compatible with the Universal Plug and Play protocols (UPNP) is necessary. On most regular home internet connections, this is a non issue. Places where enterprise networking is in place (think universities) will most likely not be able to use this program however.

    Implementing the base chatting (for LANs) was not difficult at all. A server computer opens up a socket, and a client user connects to that socket. After entering usernames, the users are able to send messages back and forth. Input is understood by the program a single character at a time. This allows for storing the unsent message by the program. If User_1 is in the middle of a message and User_2 sends a message during this time, the program sends the needed amount of backspace characters to the terminal of User_1, prints User_2's message, then prints the stored version of User_1's message back to the screen.

    I wanted the program to be able to work over the internet however as well. In thinking about this problem, I realized the immediate solution of having users open up ports on their routers manually was far from user friendly, even for the technically inclined audience the program is meant for. That is where UPNP comes into play. A library for communication with routers was already written in C, with Python bindings also being present. I figured out how to use this library, and implemented it into clChat, automating the process for the server user of figuring out their public IP, deciding on a good port to use, and manually adding the port mapping to their router.



    Wi-Fi PSK Application


    Developed using: Node.js, Angular.js, Flask, Bootstrap, JS, HTML, CSS

    I wrote UNLV's Wi-Fi PSK (Pre-Shared Key) Application in order to make an easy to use system to manage access on to UNLV's PSK net. It grants IT groups outside of the network engineering department to control user access to the net, adding and removing users by MAC address. Before this system was developed, people at UNLV only had Wi-Fi WPA-2 Enterprise as a viable way for mobile devices to access the internet. This solution sought to enable people to access a PSK network in order to get devices that aren't compabile with WPA-2 Enterprise systems online wirelessly.

    The network portion of this project was handled with the help of one of the Network Engineers at UNLV. In order for the system to be manageable, we needed some way to limit use to people permitted to use the network. We chose to do this via access restrictions based on a devices MAC address being registered. The system I developed allowed for people outside of the network team to manage the devices allowed on this net, as well as the potential for users to manage their devices themselves.

    The system itself is comprised of a few different parts. There is the front end portal for managing access to the net, a front end portal for managing the administation of the site, an authentication server for handling the checking of administrator credentials, and a back end server for handling data transfer to people managing the site.

    The front end portion of the system is a basic Bootstrap/Angular.js application. It communicates with the authentication and backend servers via a JSON REST API.



    The authentication for the site was a unique challenge for me. The system required multiple different roles for the site. At the lowest level, users needed to be able to log in and manage their own personal devices. The backend routes required to enable this feature were created, but administrative red tape prevents the deployment of it. The viable user types included admins who could add clients and devices connected to those clients, as well as super users who could add LDAP groups to check against for admin accounts. The authentication server works by checking LDAP credentials sent when a user accesses the site, ensuring first that the username sent was in a permitted AD group, then checking that the password associated with that user was correct. I chose to implement the functionality of the authentication server in node.js, based off of the libraries written that made this task easier.

    The meat of the backend is implemented in Flask. The front end communicates with this backend in order to handle all data transfer. The authentication server also communicates with it briefly in order to check group ownership of users attempting to log into the website.

    I have explained a lot here, and this project was fairly complex with a lot of moving parts. It took a few months to complete. Beyond the completion of the actual code needed to run it, working with various teams and people in the networking department was required to make it a reality.

    Convention Logistics Tracker

    My senior design project consisted of eight team members (I am third from the right!) creating a software suite called Convention Logistics Tracker. We won First Place at the 2017 Senior Design Competition for the Computer Science category. I was the lead backend developer, with the responsibility of getting the backend team up to speed on proven techniques I had given my experience. I also did devops for the team, ensuring that we had a reliable backend API that the app and frontend teams could rely on for their HTTP calls.


    The suite contains three software packages: organizer portal, exhibitor portal, and consumer app. In the organizer portal, conference organizers are able to update app information and conference details, such as floor plans, weekly/daily schedules, exhibitors lists, news, and so forth. The exhibitor portal is either filled out by a conference organizer or the exhibitor themselves. Here, exhibitors can upload a description, company logo, and news items for their own booth. For the consumer app, the app developers focused on creating an Android app that displayed all the data pulled from the Exhibitor and Organizer portal (using the REST API built by the backend team. Features of the app include News, Exhibitors List, Floor Plan, and Schedule.

    As leader of the backend team, not only did I write a good portion of the code, but also helped our other amazing backend developer get up to speed with the basics of how a REST API based backend works. By the end of it, the backend team had a very effective work pipeline working. This process consisted of us finding out exactly what the needs of the frontend and app teams were and figuring out if the database had the structure and information required to fullfill those needs. We also had to figure out the best way to execute on those needs. If the problem we faced was novel, I would tackle it or delegate if it was something easy enough to do where I could review for efficiency and correctness.


    This project was probably one of my most fun ones to date. While I have worked on plenty of projects alone or with a partner or two, this project brought together a good group of people all with varied qualities to the table to create one functional product. The process of working with a team and using effective communication in order to understand how you can best contribute your cog into the machine with many moving parts that was a project such as this was interesting to say the least. It wasn't necessarily easy, but was absolutely one of the best learning experiences I have had the opportunity to go through.
